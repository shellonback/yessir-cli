# Yessir — the safety layer for autonomous coding agents

**Let AI coding agents work. Keep the dangerous decisions yours.**

By **[ShellOnBack](https://shellonback.com)**.

Yessir is a local safety layer for terminal-based AI coding agents. It
automatically approves safe actions, blocks dangerous ones, and falls back
to you for anything in between.

Works with Claude Code, OpenAI Codex, Gemini CLI, Aider, and any
terminal-based coding agent.

- Local-first. No cloud proxy. No hosted runtime.
- Zero runtime dependencies.
- Single YAML policy file, version-controllable.
- Two integration paths:
  1. **Hook** — Claude Code calls `yessir hook` before every tool use.
     Takes effect on any Claude Code session that reads the project's `.claude/settings.json`.
  2. **Wrapper** — `yessir claude` runs the agent inside a managed PTY
     and watches live output.

## Quick start

```bash
npx yessir-cli init --hook
```

That:

1. writes `.yessir/yessir.yml` with a conservative default policy;
2. wires `.claude/settings.json` so Claude Code invokes `yessir hook`
   on every `PreToolUse` event.

Open Claude Code as usual. Safe commands (`git status`, `npm test`, etc.)
get approved instantly without you clicking anything. Dangerous ones
(`rm -rf`, `sudo`, `git push --force`) get blocked. Everything else is
escalated to you.

## How it works

```
Claude Code is about to run a tool
        |
        v
PreToolUse hook fires `yessir hook`
        |
        v
Policy engine matches the command / file write against .yessir/yessir.yml
        |
        +-- deny        -> tool is blocked
        +-- allow       -> tool is approved, no prompt shown
        +-- manual      -> Claude Code asks you (as usual)
        +-- destructive -> escalated to you even if not explicitly listed
```

Policy precedence:

1. `deny` — always wins
2. `require_manual` — always wins over allow
3. destructive heuristics (`rm -rf`, `sudo`, `curl | bash`, shell metacharacters) — escalated unless explicitly allowed
4. `allow` — approved
5. unknown — escalated to user (or routed to the optional AI reviewer in `hybrid`/`ai` mode)

## Policy file

Generated by `yessir init`. Example:

```yaml
mode: hybrid

allow:
  commands:
    - git status
    - git diff *
    - npm test
    - npm run test *
    - npm run lint
  read:
    - "**/*"
  write:
    - src/**
    - tests/**

deny:
  commands:
    - rm -rf *
    - sudo *
    - git push --force *
    - curl * | bash
    - npm publish

require_manual:
  commands:
    - git push
    - git push *
    - docker compose up *
    - npm install *

ai_reply:
  enabled: true
  model: auto
  max_tail_lines: 300
```

### Command patterns

- `*` matches any run of non-shell-metachar characters.
  So `git diff *` matches `git diff HEAD` but NOT `git diff && rm -rf .`.
- Every command is normalized (whitespace collapsed, trimmed) before matching.
- Shell metacharacters (`|`, `&`, `;`, `<`, `>`, backtick, `$`) in the command
  cause escalation unless the exact pipeline matches an allow rule verbatim.

### Path patterns

- `*` matches any run of non-slash characters.
- `**` matches any number of path segments.
- `?` matches a single non-slash character.

## Modes

| Mode     | Behavior                                                                 |
| -------- | ------------------------------------------------------------------------ |
| `quick`  | Deterministic only. No AI calls. Unknown commands escalate to the user.  |
| `ai`     | Every ambiguous decision is routed to the configured AI reviewer.        |
| `hybrid` | Default. Deterministic for known commands, AI for the rest (if enabled). |

Override at runtime with `--mode`:

```bash
yessir claude --mode quick
```

## Commands

```text
yessir init [--force] [--hook]
yessir hook                        # reads PreToolUse JSON on stdin
yessir claude [--mode ...] [--dry-run] [--no-ai]
yessir codex  [...]
yessir gemini [...]
yessir -- <command> [args...]      # wrap any interactive tool
yessir doctor                      # inspect setup
yessir explain "<command>"         # print the decision for a command
```

## AI reviewer

AutoApprove ships with a `NoopReviewer` that always escalates. To plug a real
reviewer in, implement `AiReviewer` from the package entrypoint:

```ts
import { AiReviewer, ReviewerInput, ReviewerOutput } from 'yessir';

export class MyReviewer implements AiReviewer {
  readonly name = 'openai';
  async review(input: ReviewerInput): Promise<ReviewerOutput> {
    // call your model of choice
    return { decision: 'manual', reason: 'explain why' };
  }
}
```

All terminal context passed to the reviewer is run through `redactSecrets`
before leaving the machine.

## Safety principles

1. Local-first by default. Policy stays in your repo.
2. Deny rules override everything.
3. Unknown destructive commands are never silently approved.
4. The AI reviewer can never execute commands; it only returns a decision.
5. Policy is versionable inside the repo.
6. Secrets are redacted before any AI context leaves the process.
7. The terminal renderer is not the enforcement layer — the policy engine is.

## FAQ

**Where does Yessir come from?**
Yessir is built and maintained by [ShellOnBack](https://shellonback.com). It
extracts the core safety loop that PromptOps Manager uses internally and
ships it as a standalone OSS CLI — no UI, no cloud, just the policy engine
and the hook.

**Does it send my code to a cloud service?**
No. The hook mode is entirely local. If you wire a real AI reviewer, you
control where the context goes.

**Can I use it without Claude Code?**
Yes. Use `yessir -- <any interactive tool>` to wrap any CLI in a PTY
with the same policy engine.
